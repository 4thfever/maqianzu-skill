# 马前卒 Skill 执行计划

## 1. 项目目标

### 1.1 目标定义

本项目最初的目标，是将当前仓库重构为一个可供本地 agent 使用的 `maqianzu` skill。

该 skill 的核心能力是以马前卒本人的口吻回复问题。

- 基于公开节目材料与整理后的知识库
- 按照马前卒常见的分析路径组织思考
- 使用接近其公开表达习惯的语言风格回复问题
- 在可引用时尽量说明依据来自哪些节目材料或相关主题

### 1.2 非目标

以下内容不作为本项目目标：
- 虚构其私下经历、私人信息或未公开观点
- 为任何问题强行编造“马前卒一定会这么说”的结论
- 建立复杂的外部向量数据库或在线服务

### 1.3 核心设计原则

- 以本地 agent 可直接读取的文件系统知识库为核心
- 以“索引驱动检索”代替“黑盒自动 RAG”
- 以“分析框架稳定”优先于“口吻模仿相似”
- 以可重复构建的工具链代替手工整理
- 以公开可维护的仓库结构代替临时堆文件

## 2. 当前仓库现状判断

当前仓库来自 `therealXiaomanChu/ex-skill` 的复制版本，仍保留明显的原模板痕迹：

- `SKILL.md` 仍然是“前任 skill 创建器”的触发逻辑与工具说明
- `README.md` 仅做了表层替换，内容与当前目标不一致
- `INSTALL.md`、`docs/PRD.md`、`docs/COMPLIANCE.md`、`prompts/session_summary.md` 等内容与目标不匹配
- `tools/` 目录为空，尚未建立可复用的数据处理流水线

结论：本项目应视为“保留基础 skill 仓库壳子后的重建”，而不是“小修小改”。

## 3. 总体实施路径

整个项目分为 6 个阶段执行：

1. 仓库重构与骨架铺设
2. Skill 行为与回答协议设计
3. 语料来源整理与过滤
4. Chunk 规范制定与切分实现
5. 检索索引与知识目录生成
6. 发布前整理与文档完善

每个阶段都应产出可审阅结果，避免在所有内容完成后一次性验收。

## 6. 阶段二：Skill 行为与回答协议设计

### 6.1 目标

定义 skill 的“脑子”和“边界”，确保之后加入语料时有统一行为规范。

这一阶段不直接凭印象填写 `prompts/` 文件，而是先完成一轮尽可能广泛的外部信息搜集和归纳，再落为正式文档。

### 6.2 需要产出的核心文件

#### `prompts/persona.md`

用于总结公开表达层面的风格特征，例如：

- 语言节奏
- 常见句式
- 论证语气
- 面对荒谬现象时的表达风格
- 对宏观问题与具体执行问题的切换方式

该文件应强调“接近公开表达方式”，而不是“模仿本人身份”。

#### `prompts/analysis_framework.md`

用于定义“马前卒式分析”的工作方法。计划包含以下分析框架模块：

- 先界定问题的结构性背景
- 识别利益相关方与激励机制
- 检查制度设计与执行条件
- 区分口号、目标与实际约束
- 评估现实成本、资源分配与次生后果
- 适当进行历史类比或制度比较
- 最后给出具有可操作性的判断或批评

#### `prompts/response_policy.md`

用于定义回答边界与输出规则，计划包含：

- 不虚构私人经历或未公开消息
- 在证据不足时明确说明不确定性
- 能引用时尽量说明依据来自哪些节目材料或相关主题
- 没有直接语料时允许做框架性推演，但必须说明属于风格化分析，不等于原话依据

#### `prompts/retrieval_workflow.md`

用于告诉 skill 在回答时如何读取知识文件，流程预期为：

1. 识别问题主题
2. 优先查看 `knowledge/index.md`
3. 再查看相关 `knowledge/topics/*.md`
4. 最后按需读取具体 `chunk` 文件
5. 基于有限上下文作答，不盲目扫描大量文件

### 6.3 研究先行原则

Phase 2 的核心原则是“研究先于填写”。

在 `prompts/persona.md`、`prompts/analysis_framework.md`、`prompts/response_policy.md` 真正写入具体内容之前，必须先完成尽可能广泛的互联网信息搜集，并把搜集结果转化为可验证、可归类、可追溯的研究笔记。

原因如下：

- 马前卒的公众形象并不只来自单一节目
- 仅靠个人印象很容易把“某几期节目感觉”误写成稳定风格
- 只看支持者总结会失真，只看批评者总结也会失真
- 需要区分“高频特征”“偶发特征”“争议印象”“外界标签”

### 6.4 信息搜集范围

为保证后续文件填写尽量稳妥，Phase 2 计划将信息来源分为四类。

#### A. 第一手公开材料

优先级最高，应作为人格、方法论与风格总结的主要依据。

计划重点覆盖：

- 《睡前消息》节目正文与转录
- 直播相关公开内容
- 马前卒本人公开发布的问答、说明、访谈与长文
- 与其工作方式、节目取材、论证习惯有关的自述材料

使用目标：

- 提炼高频用语
- 提炼稳定分析路径
- 提炼对不同问题的常见切入方式
- 提炼回答中的常见结构和节奏

#### B. 第二手整理材料

作为补充来源，用于帮助定位线索，不直接作为最终结论依据。

计划覆盖：

- 百科条目
- 节目存档站整理
- 观众制作的索引、合集、摘要
- 对其节目形式、常见议题、工作方式的整理页面

使用目标：

- 帮助快速建立时间线与栏目轮廓
- 帮助发现值得重点回看的主题和阶段
- 帮助补全背景信息

#### C. 外界评价材料

包括支持者、批评者和中性观察者的评论，用于识别公众感知中的风格标签。

计划覆盖：

- 对节目风格的评价
- 对其论证方式的批评
- 对其立场和表达习惯的争议点
- 对其“高频特点”的外界总结

使用目标：

- 识别哪些特征是外界普遍感知到的
- 识别哪些印象存在明显争议
- 避免把争议标签直接写成稳定人格设定

#### D. 节目外补充材料

作为侧面佐证，帮助判断其表达方式在不同语境下是否稳定。

计划覆盖：

- 访谈
- 问答节目
- 其他公开平台发言
- 不同时期的公开文章

使用目标：

- 比较其在节目内外的表达差异
- 判断哪些特征是节目体裁带来的，哪些是个人长期风格

### 6.5 信息搜集方法

这一阶段建议按以下顺序展开：

1. 建立资料清单
2. 标记来源类型
3. 区分第一手与第二手
4. 对每条材料提取观察点
5. 归并为稳定结论、争议结论和待观察结论

每次记录一条材料时，至少要整理以下字段：

- 来源标题
- 来源链接
- 来源类型
- 发布时间或对应节目期次
- 观察到的风格特征
- 观察到的分析路径
- 是否适合直接进入 prompt 结论

### 6.6 研究输出文件建议

为了避免直接把杂乱素材塞进正式 prompt，建议在正式填写前先新增研究型中间文件。

建议新增：

- `docs/dev/research-persona.md`
- `docs/dev/research-analysis-framework.md`
- `docs/dev/research-style-observations.md`
- `docs/dev/research-source-log.md`

职责建议如下：

`docs/dev/research-source-log.md`

- 记录搜集到的外部来源
- 标明来源类型与可信度
- 标明是否已阅读、是否已提炼

`docs/dev/research-style-observations.md`

- 记录具体的说话方式、语气习惯、句式特征
- 区分高频特征与偶发特征

`docs/dev/research-analysis-framework.md`

- 记录其常用分析路径
- 区分“对结构的判断”和“对态度的表达”

`docs/dev/research-persona.md`

- 记录可稳定沉淀为 persona 的部分
- 排除明显依赖具体事件的短期姿态

### 6.7 Prompt 文件的填写规则

在完成研究型中间文件前，不应直接大段填写正式 prompt。

正式填写时建议采用以下规则：

- `prompts/persona.md` 只写高频且跨来源稳定的表达特征
- `prompts/analysis_framework.md` 只写可重复观察到的分析路径
- `prompts/response_policy.md` 要吸收“信息不足时如何收口”的研究结论
- 对存在争议但难以确认稳定性的特征，先留在研究文件，不进入正式 prompt

### 6.8 重点研究问题

为了让 Phase 2 的搜集更有方向，建议重点回答以下问题：

1. 他分析问题时，是否总是先找结构性原因，再谈个人责任
2. 他在讨论社会、经济、产业、国际问题时，是否有共通的推理骨架
3. 他表达批评时，更常用讽刺、设问、类比，还是直接结论
4. 他在节目中如何处理“信息不足但必须判断”的场景
5. 他如何把宏观制度问题和具体执行问题连接起来
6. 哪些语气特征是稳定的，哪些只是节目情绪带来的暂时性表现
7. 外界如何概括他的风格，这些概括和第一手材料是否吻合

## 7. 阶段三：语料来源整理与过滤

### 7.1 目标

从 `bedtimenews/bedtimenews-archive-contents` 中提取仅与目标相关的内容，并将其转为本仓库可用的中间格式。

### 7.2 数据来源范围

只处理以下两类内容：

- `main`
- `livestream`

不纳入其他非目标内容，以保持风格和主题边界稳定。

### 7.3 处理原则

- 保留节目标题、日期、来源路径等基础信息
- 尽量保留完整文本结构，避免一开始过度压缩
- 对明显无效的包装性文本、导航性文本、噪音内容进行清理
- 保持原文事实内容不被误改

### 7.4 计划脚本

#### `tools/fetch_sources.py`

职责：

- 拉取或同步上游语料仓库
- 仅复制需要的目标目录到本地工作区

#### `tools/normalize_archive.py`

职责：

- 统一文件编码
- 统一命名风格
- 去除无关结构噪音
- 为后续切分准备标准化文本

#### `tools/utils.py`

职责：

- 提供路径工具、元数据提取工具、文本清洗工具等基础函数

### 7.5 阶段产物

- 一套标准化的中间语料
- 明确、可复现的抓取与清洗流程

### 7.6 验收标准

- 所有导入材料均来自 `main` 或 `livestream`
- 文件编码、命名、目录风格统一
- 可以在不手工干预的前提下再次运行脚本复建中间数据

## 8. 阶段四：Chunk 规范制定与切分实现

### 8.1 目标

把清洗后的节目内容切成适合本地 agent 读取与检索的知识单元。

### 8.2 核心判断

本地 agent 更适合“读结构化文件并按索引逐步展开”，而不是依赖一个不可见的自动向量检索层。因此 chunk 的目标不是服务黑盒 RAG，而是服务：

- 文件级定位
- 主题级索引
- 有限上下文精读
- 明确引用来源

### 8.3 切分原则

采用“按论证单元切分”的策略，而不是纯粹按固定字数切分。

每个 chunk 应尽量满足：

- 聚焦一个核心议题
- 保留主要结论
- 保留必要推理链
- 避免把多个独立议题混在一起
- 控制在 agent 适合快速读取的体量内

### 8.4 建议的 chunk 长度

建议范围：

- 常规 chunk：约 500 到 1500 中文字
- 极短材料可适度低于该范围
- 涉及完整论证链的复杂材料可适度高于该范围

说明：

- 不以固定长度为第一原则
- 以“读完后仍能保留完整论证”的边界为准

### 8.5 chunk 文件格式

每个 chunk 建议使用统一 markdown 模板：

```md
---
title: 节目标题
date: 2023-01-01
source_type: main
episode_id: main-2023-01-01-001
topics:
  - economy
  - governance
keywords:
  - 地方财政
  - 房地产
summary: 关于地方财政依赖房地产的结构性风险分析。
source_path: upstream/main/2023/xxx.md
---

## 核心论点

...

## 原始内容整理

...
```

### 8.6 文件命名规则

文件名应尽量语义化，避免无意义流水号。建议格式：

```text
YYYY-MM-DD_source_topic-keyword_01.md
```

示例：

```text
2023-05-12_main_local-finance_housing-risk_01.md
```

### 8.7 计划脚本

#### `tools/split_chunks.py`

职责：

- 根据清洗后的文本切分论证单元
- 输出标准 chunk 文件
- 为每个节目生成配套 `meta.md`

### 8.8 节目级配套文件

每个节目目录建议至少包含：

- `meta.md`
- `chunk-*.md`

其中 `meta.md` 应包括：

- 节目标题
- 日期
- 来源类型
- 主题标签
- 节目摘要
- chunk 列表
- 适用问题提示

### 8.9 阶段产物

- 一组结构化、可读、可检索的 chunk 文件
- 节目级 `meta.md`

### 8.10 验收标准

- 任意单个 chunk 可以被独立读取并理解主要论点
- 任意节目内容都能通过 `meta.md` 找到对应 chunks
- 文件名与元数据足以支持 agent 进行主题定位

## 9. 阶段五：检索索引与知识目录生成

### 9.1 目标

为本地 agent 构建“先看地图，再读正文”的知识读取路径。

### 9.2 总体策略

采用双层检索结构：

- 第一层：主题索引与总索引
- 第二层：具体 chunk 精读

### 9.3 必需索引文件

#### `knowledge/index.md`

作用：

- 提供整个知识库总览
- 告诉 agent 各主题、各目录、各来源的分布情况
- 指引优先读取路径

- 常用起始读取路径

- 在具体回答前帮助 Claude 选择最相关材料

## 11. 本地 Agent 检索适配策略

### 11.1 关键判断

本项目不应假设运行环境自带成熟的黑盒式 RAG 系统。

更合理的理解是：

- agent 可以读取文件
- 可以根据 skill 指令按路径和文件名查找
- 可以先读索引，再读正文
- 可以在目录结构良好的情况下进行有效材料定位

因此，本项目知识库设计的关键不在“是否切成 chunk”本身，而在于：

- 是否有总索引
- 是否有主题索引
- 是否有清晰文件名
- 是否有统一 metadata
- 是否明确告诉 skill 先读哪里

### 11.2 推荐读取流程

预期技能使用流程如下：

1. 用户提问
2. agent 识别问题所属主题
3. 读取 `prompts/analysis_framework.md`
4. 读取 `prompts/response_policy.md`
5. 读取 `knowledge/index.md`
6. 读取相关 `knowledge/topics/*.md`
7. 按需读取少量 `chunk` 文件
8. 生成回答，并在可行时说明依据

### 11.3 设计结论

结论如下：

- 必须做 chunk
- 但不能只做 chunk
- 必须配套索引、主题文件、命名规范与元数据
